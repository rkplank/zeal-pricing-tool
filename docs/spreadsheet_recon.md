## 1. Column mapping in `PricingSheet`

The header row is misleading — column F is labeled "Buy" but actually computes the online sell price. Mapping to the algorithm spec's four output names:

| Spec output | Column | Notes |
|---|---|---|
| `online_sell` | **F** | Mislabeled "Buy" in header; formula `=B - ebay_differential` confirms it's the storefront listing price |
| `in_mail_buy` | **C** | Header "In-Mail" |
| `in_store_buy` | **E** | Header "In-Store" |
| `electronic_buy` | **D** | Header "Electronic" |

Other relevant columns:
- **A** — merchant display name (also section divider labels — see §4)
- **B** — eBay sell percentage (input). Hardcoded number for normal merchants, `"N/A"` or `"No Data"` text for the patterns in §4
- **G** — tier label: `T24`, `C`, `Z`, or `NC` (descriptive only per algorithm spec §3)

## 2. Canonical formula structure

For a normal merchant (e.g. Mastercard, row 3) with all four channels active:

```
F (online_sell)    = B - InputsandMargins!Bxx     ← xx is B23 or B24
C (in_mail_buy)    = F - <in_mail_margin> - <paypal> - <in_mail_bad_debt> - <postage>
E (in_store_buy)   = F - <in_store_margin> - <paypal> - <postage> - <in_store_bad_debt>
D (electronic_buy) = C - <e_bonus>
```

Worked Mastercard values (row 3): B3=1.05 → F3=1.005 → C3=0.895, E3=0.647, D3=0.815.

The formula structure matches algorithm spec §5 exactly. The novelty is in *which row of `InputsandMargins`* each formula references — see §3.

### 2.2 Differential decomposition

Both differentials follow the same structure: eBay-side costs minus the storefront-side costs and pass-through bonus.

- `B23 = B2 + B4 − B6 − B5 = 13% + 1% − 6.5% − 3% = 4.5%` (competitive/T24)
- `B24 = B2 + B4 − B7 − B5 = 13% + 1% − 8.5% − 3% = 2.5%` (Zen-Only/NC)

Same formula shape; the only difference is which "online sell bonus" gets passed to buyers — 6.5% in competitive markets vs 8.5% where Zeal has less competition (a wider markdown closes the gap with competitor pricing).

## 3. Per-merchant config drift — confirmed

Algorithm spec §3 predicted that merchants within the same tier reference different margin rows. The full audit (now in the `MarginRefs` tab of the workbook, generated by the recon) confirms this. Key examples:

- **Home Depot (row 11, tier `C`)** — mixes tiers. In-store pulls **B11 (T24)**, in-mail pulls **B14 (Competitive)**. Every other normal merchant uses a single tier consistently for both. This is the canonical example of why per-merchant config is necessary.

- **Bankrupt section (rows 280–283)** — uses a mix of `B13/B14` and `B15/B16` references even though all are flagged dead. Likely stale config from before they were marked bankrupt; not deliberate.

- **Local NC merchants (rows 250–274, 294)** — uniformly use `{B17, B18}` plus `B21` for e-bonus where electronic is offered.

The `MarginRefs` audit tab in the workbook is the source of truth for which row each merchant references for each of the four computed columns. The parser reads from formulas directly, but the audit tab is useful for spot-checking parser output.

## 4. Row classification — five types of rows

The parser must classify every row in `PricingSheet` before extracting config. The five types:

### 4.1 Type "normal"
Standard merchants with formula-driven prices in all four output columns.
- A: merchant name (string)
- B: eBay sell pct (number)
- C, D, E, F: formulas of the canonical shape in §2
- G: tier label
- **Action:** parse config from formulas, include in baseline.

### 4.2 Type "channel_ineligible"
Merchant where one or more channels are deliberately disabled.
- The disabled channel cell contains the literal string `"No"` (plain text, not a formula).
- Active channels still have normal formulas.
- Convention applies to C, D, E uniformly (not just D as the algorithm spec implied).
- **Examples:** the merch-credit Home Depots (rows 12, 13) have `D = "No"`; row 14 has `C = "No"` and `E = "No"` while `D` is hardcoded.
- **Action:** parse the active channels normally; record disabled channels via flags.

### 4.3 Type "no_ebay_data_local" (Pattern A)
Local / small-volume merchants where eBay has no useful sold-listing data, but Zeal still sells the card. Convention is *inverted*:
- B: literal text `"N/A"` (no formula)
- F (online_sell): **hardcoded number** (e.g. 0.65, 0.75, 0.85, 0.95) — F is the manual input
- C, E: standard formulas referencing F (they compute correctly)
- D: either `"No"` or `=C-B21` (the "E Bonus Special" 17%)
- G: tier `NC`
- **Examples:** row 250 Andiamo, row 270 Plum Market, row 274 Zingerman's, row 294 M Den.
- **Action:** treat `online_sell` as direct config (`online_sell_override`) rather than a derived value.

### 4.4 Type "bankrupt_broken" (Pattern C)
Bankrupt merchants where someone replaced B with text rather than restructuring the row. Formulas still try to compute and propagate `#VALUE!`.
- B: `"N/A"` or `"No Data"` (inconsistent)
- F has `=B-B23` formula → cascades `#VALUE!` to C, D, E
- All four outputs are `#VALUE!`
- **Examples:** row 280 Avenue, row 281 Bebe, row 283 Carson's. (Row 282 Big Lots in the same section still computes — different state.)
- **Action:** detect, log, exclude from baseline. Not a real merchant for v1.

### 4.5 Type "section_divider"
Non-merchant rows used to label sections of the sheet. Two detection patterns:

**Name-keyed dividers** — A contains a known section label:
- Literal strings: `"PHYSICAL ONLY"`, `"Bankrupt"`, `"Merchant"` (the original header repeated mid-sheet)
- Examples: row 249 (`PHYSICAL ONLY`), row 278 (`Bankrupt`).

**Formula-keyed aggregates** — F column contains `=AVERAGE(...)`:
- Operator-inserted summary rows showing section averages for visibility, not merchant pricing.
- F-column `=AVERAGE(...)` formula → treat identically to name-keyed dividers.
- Examples (all reclassified 2026-05-02):
  - Row 19: `BREAD AND BUTTER` — `=AVERAGE(F3:F18)`
  - Row 60: `TOP CARDS` — `=AVERAGE(F21:F59)`
  - Row 83: `COVID WATCH LIST` — `=AVERAGE(F62:F82)`
  - Row 210: `TAKE ONLINE` — `=AVERAGE(...)`
  - Row 276: `LOCAL` — `=AVERAGE(...)`

**Action:** skip entirely. No merchant_id, no config, excluded from baseline.

### 4.6 Classification logic (sketch for parser)

**F-column is the source of truth for row type** because the downstream channels (C, D, E) compute from F, not B. B-column values can be stale or misleading — notably, row 253 Biggby Coffee has a numeric B (0.668) but F is a hardcoded literal (0.75), making it a Pattern A merchant that the old B-keyed rule mis-classified as "normal". See decisions_log.md 2026-05-02.

```python
def classify_row(name_val, b_val, f_val, f_formula):
    # f_val: cached value from data_only workbook
    # f_formula: raw cell value from formula workbook (formula str or literal)
    if not isinstance(name_val, str) or not name_val.strip():
        return "skip_blank"
    if name_val.strip() in {"Merchant", "PHYSICAL ONLY", "Bankrupt"}:
        return "section_divider"
    if isinstance(f_formula, str) and f_formula.startswith("=AVERAGE("):
        return "section_divider"   # operator-inserted aggregate row
    if isinstance(f_formula, str) and f_formula.startswith("="):
        # F is a real formula; normal if it computed, bankrupt_broken if it errored
        if isinstance(f_val, (int, float)):
            return "normal"
        return "bankrupt_broken"
    if isinstance(f_formula, (int, float)):
        return "no_ebay_data_local"   # F is a hardcoded literal (Pattern A)
    return "bankrupt_broken"
```

Channel-ineligibility (`"No"` in C/D/E) is orthogonal to row type — a "normal" row can still have one or more channels marked `"No"`. Detect after classification.

## 5. Encoding conventions (cheat sheet)

| Marker | Meaning | Where it appears |
|---|---|---|
| Plain text `"No"` | Channel deliberately disabled for this merchant | Cells C, D, or E |
| Plain text `"N/A"` | Either Pattern A (eBay has no data, F is manual) or Pattern C (bankrupt). Disambiguate via F. | Cell B |
| Plain text `"No Data"` | Same as `"N/A"` but inconsistent label. Treat identically. | Cell B (Carson's only, row 283) |
| `#VALUE!` error | Cascade from non-numeric B; bankrupt-broken row | C, D, E, F |
| Blank cell | Section divider row, or empty | Throughout |
| Hardcoded number | (1) `F` for Pattern A merchants; (2) `D14` Home Depot eStore Credit = 0.65 | F (Pattern A); D14 (one-off) |

- `home_depot_merch_credit_no_id` (row 12)

- `home_depot_estore_credit` (row 14)
- `menards_rebate` (row 16) — uses `_rebate` qualifier rather than `_merch_credit` to preserve the display-name distinction
- `lowes_merch_credit_no_id` (row 18) — keeping `no_id` even though Lowe's has only this one variant; future-proofs if a `with_id` is added
- `ikea_merch_credit` (row 30)
- `target_merch_credit` (row 51)
- `tjm_homegoods_marshalls_merch_credit` (row 53) — parent merchant slug also needs disambiguation; consider whether to treat as one combined merchant or three. For v1, treat as single combined merchant matching the spreadsheet.

### 7.4 `merch_credit_variant` flag detection rule

Set `merch_credit_variant = True` for rows whose display name matches any of these substrings (case-insensitive):

- `merch credit`
- `merchandise credit`
- `estore credit`
- `rebate`

Rationale: all four are "non-standard variants priced separately" per the spec definition. The Menards Rebate pattern matches a merch-credit's pricing structure exactly, so it's appropriate to flag it the same way for v1. Future versions can split if behavior diverges.

## 8. Spec gaps surfaced by the recon

Findings the algorithm spec didn't originally cover. **All resolved** by patches to `pricing_algorithm.md` and `architecture.md` — see those documents and `decisions_log.md`.

### 8.1 Per-channel ineligibility (not just electronic)
**Resolution:** `MerchantConfig` extended from one boolean (`electronic_eligible`) to three (`in_store_eligible`, `in_mail_eligible`, `electronic_eligible`). Engine's per-channel computation guards on each one, returning `"No"` sentinel where ineligible.

### 8.2 Pattern A merchants — hardcoded `online_sell`
**Resolution:** `online_sell_override: float | None` field added to `MerchantConfig`. When set, engine uses it directly instead of computing from eBay input. ~25 NC-tier merchants use this.

### 8.3 Hardcoded `electronic_buy` (row 14)
**Resolution:** `electronic_buy_override: float | None` field added to `MerchantConfig`. When set, engine uses it directly instead of computing from `in_mail_buy - e_bonus`. Currently used only by Home Depot eStore Credit.

### 8.4 `ebay_differential` is a formula, not a constant
**Resolution:** v1 stores the differential as a per-merchant value (faithful to spreadsheet). The components (`online_sell_bonus_competitive` = 0.065, `online_sell_bonus_zen_nocomp` = 0.085) added to global constants for completeness and v2 use.

## 9. Implications for parser design

Concrete decisions baked in by the recon:

1. **Two-pass openpyxl read** is correct — formulas drive config extraction, computed values drive baseline verification.
2. **Row classification happens first**, before any config extraction. Use the §4.6 sketch.
3. **Per-channel ineligibility** is a separate flag set, detected after row classification.
4. **`MarginRefs` audit tab** in the workbook is the spot-check companion. Parser output should match it row-for-row for the cell-reference column.
5. **Pattern A merchants populate `online_sell_override`** rather than `ebay_sell_pct`-driven config.
6. **Bankrupt-broken rows are documented exclusions**, not failures. The parser report should list them by row number and merchant name.
7. **Section dividers and blank rows are silent skips** — no logging needed beyond a count.
8. **Merch-credit variant flag** uses the §7.4 substring rule.
9. **Total expected baseline record count:** 295 rows total minus 7 dividers (8, 249, 278, 19, 60, 83, 210, 276) minus 4 bankrupt-broken (275, 280, 281, 283) minus 4 documented exclusions (44, 45, 46, 70) ≈ 280 baseline records. Confirm exact count from extraction report.

The parser report (from the parser session) should produce:
- Row-by-row classification (normal / no_ebay_data_local / channel_ineligible / bankrupt_broken / section_divider / skip_blank)
- For each `normal` and `no_ebay_data_local` row: extracted config + spreadsheet's computed values + engine's recomputed values + diff
- Total counts per type
- A "spec deviations" log: any merchant whose config doesn't match its tier's expected references (this is expected per algorithm spec §3, but the report quantifies the drift).

## 11. Tier-vs-margin drift audit

The extraction report's "Tier-vs-margin-row audit" flags every merchant whose in-store or in-mail formula references a margin row that doesn't match the expected row for their tier label. This section records what those flags mean so the context is preserved for the v2 margin audit (spec §10.10).

**Margin values for reference** (InputsandMargins B11–B18):

| Tier | In-store margin (row) | In-mail margin (row) |
|------|-----------------------|----------------------|
| T24  | 25.0% (B11)           | 3.0% (B12)           |
| C    | 25.0% (B13)           | 7.0% (B14)           |
| Z    | 29.0% (B15)           | 11.0% (B16)          |
| NC   | 30.0% (B17)           | 15.0% (B18)          |

**Important:** T24 and C share the same in-store margin (25.0%). Any audit flag for "C uses T24-in-store" or "T24 uses C-in-store" is a cell-reference difference with zero pricing impact.

### 11.1 Value-neutral drift (T24 ↔ C in-store, both 25%)

These merchants reference B11 where B13 is expected, or vice versa. No price difference. Probably arises from the operator copying formula rows across tier boundaries without adjusting the reference.

**C using T24-in-store (B11 instead of B13):** rows 17 (lowe_s), 20 (best_buy), 32 (kohl_s), 40 (nordstrom), 41 (office_depot_officemax), 50 (target), 52 (tj_maxx_homegoods_marshalls), 56 (victoria_s_secret), 58 (walmart), 279 (bed_bath_beyond).

### 11.2 Systematic Z→C in-store drift (~100 merchants, probably deliberate)

Nearly every Z-tier merchant (rows 84–247 and a handful of post-249 rows) references B13 (C-in-store = 25%) instead of B15 (Z-in-store = 29%). The scale of this — essentially the entire Z section — makes accidental drift implausible. Most likely the operator decided at some point to price Z merchants at competitive in-store margins, and the tier label became descriptive only.

A few Z merchants are exceptions worth noting:
- Row 30 (ikea_merch_credit, Z-tier): uses C-in-store like the rest of the Z section.
- Row 282 (big_lots), 285 (christopher_banks), 289 (hooters), 290 (joann), 295 (old_country_buffet), 298 (rue_21), 299 (rite_aid): post-section Z rows, also use C-in-store.

**v2 action:** confirm with operator that Z→C in-store is intentional. If yes, update the Z tier default in the seeder to 25% and document. If no, these margins will need individual review.

The cluster of rows 61–82 was under a "COVID WATCH LIST" section header — it's plausible the operator deliberately set these merchants to competitive pricing during a period of uncertainty about their viability.
